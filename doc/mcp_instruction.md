# Интеграция MCP для LLM-сервиса

**Endpoint MCP (SSE):** `https://ibmdev.pravo.tech/sse`  
**Протокол:** MCP over SSE  
**Цель:** безопасная аналитика DWH через LLM с ограничением по схеме `ai_bi`.

## 1) Что нужно подключить в скриптах

- Подключить MCP-клиент к SSE endpoint `https://ibmdev.pravo.tech/sse`.
- Использовать стандартные MCP tools PostgreSQL:
  - `list_schemas`
  - `list_objects` (таблицы/колонки)
  - `execute_sql`
- Работать в read-only режиме (никаких `INSERT/UPDATE/DELETE/DDL`).

## 2) Обязательные ограничения (важно)

- Разрешенная схема: только `ai_bi`.
- Перед выполнением SQL:
  - проверять, что все таблицы в запросе из `ai_bi`;
  - отклонять запросы с `CROSS JOIN`;
  - отклонять потенциально тяжелые запросы без ограничений.
- Для больших выборок всегда добавлять `LIMIT` (например, 100/500 для просмотра).
- Не использовать `SELECT *` на широких таблицах без необходимости.
- Желательно выставлять timeout на выполнение запроса (на стороне сервиса).

## 3) Как LLM должна работать со структурой

1. Сначала получить метаданные (`list_schemas`, `list_objects`).
2. Собрать контекст по таблицам:
   - имя таблицы,
   - описание таблицы (если есть),
   - имя колонки,
   - описание колонки (если есть),
   - тип данных.
3. При выборе таблиц/полей приоритет:
   1) description/comment,  
   2) если описания нет — имя таблицы/поля.

## 4) Особенности данных (денормализация и дубли)

- Таблицы в основном денормализованные, возможны дубли по бизнес-ключам.
- LLM должна явно учитывать grain (уровень детализации) таблицы.
- При агрегациях:
  - использовать `COUNT(DISTINCT ...)`, где это нужно по смыслу;
  - избегать слепого `DISTINCT` везде, а применять осознанно;
  - перед `JOIN` проверять кардинальность и риск мультипликации строк.
- Для фактов и измерений по возможности соединять через понятные ключи и фильтры периода.

## 5) Рекомендуемый pipeline в LLM-сервисе

- Шаг 1: интерпретация вопроса пользователя.
- Шаг 2: подбор релевантных таблиц по именам + description таблиц/колонок.
- Шаг 3: генерация SQL с защитными правилами (без `CROSS JOIN`, с `LIMIT`/фильтрами).
- Шаг 4: (опционально) `EXPLAIN` и упрощение тяжелых мест.
- Шаг 5: выполнение `execute_sql`.
- Шаг 6: ответ пользователю + короткие допущения по качеству данных (дубли/гранулярность).
