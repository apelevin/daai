## 1. Анализ коммуникации в канале

### Что работает хорошо
- Юлия задаёт чёткое стратегическое направление (Extra Time Dashboards)
- Участники отзываются и готовы контрибьютить
- Павел Петрин сразу обозначил требования data-команды (источники + методология)
- Станислав дал глубокое описание "Карточек работы"

### Проблемы коммуникации агента (Финист Ясный)

**1. Избыточные статус-апдейты после каждого ответа**
Агент публикует полный статус (кто подтвердил, кто ждёт, следующие шаги) после каждого сообщения участника. Это создаёт "шум":
- Станислав написал развёрнутый ответ → агент ответил стеной с emoji-заголовками и перечнем всех участников
- Александр подтвердил участие → опять полный статус со всеми 8 участниками

*Рекомендация*: Краткое подтверждение (1-2 строки) + следующий вопрос. Полный статус — только по запросу или при завершении этапа. Правило уже есть в промпте (п.5), но агент его не соблюдает.

**2. Агент отвечает на методологические вопросы сам**
Михаил спросил: "Что мы хотим сделать? Что такое карточки работы?" — агент ответил сам подробно. По правилам (п.4 system_full.md) он должен был адресовать вопрос Юлии как владельцу контракта.

*Рекомендация*: Усилить в промпте правило — при вопросе участника "что это?" или "зачем?" переадресовывать владельцу контракта с чётким тегом.

**3. Двойные ответы на одно сообщение**
Когда Алексей написал про Product инстанс, агент ответил два раза подряд с практически одинаковым содержанием (строки 370-416 в переписке).

*Рекомендация*: Добавить дедупликацию ответов — если агент уже ответил в треде за последнюю минуту, не отправлять повторно.

**4. Слишком много emoji-маркеров**
Почти каждый абзац начинается с emoji (🎯, 📋, 📊, ✅, 🚀). Это противоречит правилу п.6 в system_full.md.

*Рекомендация*: В промпте уже есть запрет emoji перед заголовками, но агент обходит его, делая emoji-маркированные строки без `##`. Нужно ужесточить: "максимум 2-3 emoji на сообщение, только ✅ и ⏳ для чек-листов".

**5. Агент не задаёт уточняющие вопросы по данным**
Павел написал "Требуется согласованные источники данных (объекты - поля - фильтры)" — агент сказал "отлично!" вместо того чтобы начать прорабатывать конкретные таблицы из DWH (схема `ai_bi`).

*Рекомендация*: Интегрировать MCP-запросы к PostgreSQL прямо в процесс создания контракта (см. раздел 3).

## 2. Улучшения агента — коммуникация

### 2.1 Уменьшить "шумность" ответов

**Текущее поведение**: После каждого сообщения участника — полный статус-апдейт со всеми участниками.

**Целевое поведение**:
```
Участник: "Готов участвовать, данные из Jira"
Агент: "Записал, спасибо! Уточни — данные по Lead Time доступны через API или только выгрузка?"
```

**Изменения в `system_full.md`**:
- Добавить жёсткое правило: "Ответ на сообщение участника — максимум 5 строк. Полный статус — только по команде 'статус' или при завершении этапа."
- Удалить шаблон с перечислением всех участников после каждого ответа.

### 2.2 Переадресация методологических вопросов

**Текущее**: Агент отвечает сам.
**Целевое**: `@Юлия, Михаил спрашивает — что такое "Карточки работы"? Можешь пояснить для команды?`

### 2.3 Проактивная проработка источников данных

**Текущее**: Агент фиксирует "нужны источники" и ждёт.
**Целевое**: Агент сам запрашивает структуру DWH через MCP, находит релевантные таблицы и предлагает маппинг.

## 3. Интеграция MCP для автоматической проработки витрин

### 3.1 Автоматический маппинг контракт → витрина данных

Когда участник заполняет секцию "Источник данных" в контракте, агент должен:

1. **Запросить структуру** через MCP (`list_objects` для схемы `ai_bi`)
2. **Найти релевантные таблицы** по ключевым словам из определения контракта
3. **Показать участникам** найденные таблицы и поля
4. **Если таблиц нет** — сформировать задание на создание витрины

### 3.2 Шаблон задания на создание витрины данных

Когда контракт согласован, но витрина ещё не создана, агент генерирует документ:

```markdown
# Задание на создание витрины данных

## Контракт-основание
- ID: {contract_id}
- Название: {contract_name}
- Статус: Согласован
- Дата согласования: {agreed_date}

## Бизнес-требования
### Определение метрики
{из секции "Определение" контракта}

### Формула расчёта
{из секции "Формула" контракта}

### Гранулярность
{из секции "Гранулярность" контракта}

### Что включается
{из секции "Включает" контракта}

### Что исключается
{из секции "Исключения" контракта}

## Технические требования
### Целевая схема
`ai_bi`

### Существующие таблицы-источники
{результат MCP list_objects — таблицы, которые могут быть источником}

| Таблица | Колонка | Тип | Описание | Релевантность |
|---------|---------|-----|----------|---------------|
| ai_bi.xxx | field1 | varchar | ... | Источник для поля X |
| ai_bi.yyy | field2 | int | ... | Агрегат для формулы Y |

### Предлагаемая структура витрины
```sql
CREATE TABLE ai_bi.dm_{contract_id} (
    -- dimensions
    report_date DATE NOT NULL,
    -- ...измерения из гранулярности контракта

    -- measures
    -- ...метрики из формулы контракта

    -- metadata
    loaded_at TIMESTAMP DEFAULT NOW()
);
```

### SQL-запрос для наполнения (черновик)
```sql
-- Псевдо-SQL из контракта, адаптированный к реальным таблицам
SELECT ...
FROM ai_bi.xxx
JOIN ai_bi.yyy ON ...
WHERE ...
GROUP BY ...
```

## Валидация
### Проверки качества данных
- [ ] COUNT(*) > 0 за последний период
- [ ] Нет NULL в ключевых полях
- [ ] Значения метрики в разумном диапазоне
- [ ] Сверка с ручным расчётом за контрольный период

### Приёмочные критерии
- Витрина обновляется с частотой из "Гранулярность"
- Результат совпадает с формулой из контракта
- Ответственный за данные ({data_owner}) подтвердил корректность

## Ответственные
- Заказчик: {из "Ответственный за расчёт"}
- Исполнитель: Data-команда (@Павел Петрин)
- Проверка: {из "Ответственный за данные"}

## Связанные контракты
{из секции "Связанные контракты"}
```

### 3.3 Автоматическая проверка доступности данных

При сохранении черновика (`save_draft`) — если указан источник данных:
1. Вызвать MCP `list_objects` для схемы `ai_bi`
2. Матчить таблицы по ключевым словам из источника
3. Если таблицы найдены — записать в контракт: `Состояние данных: Данные есть, таблицы: ai_bi.xxx, ai_bi.yyy`
4. Если НЕ найдены — записать блокер: `Витрина не создана, требуется задание для Data-команды`

Это уже частично реализовано (п.7 в "Инициация нового контракта"), но не доведено до автогенерации задания.

## 5. Новые фичи для канала

### 5.1 Команда `/витрина {contract_id}`

Участник пишет `/витрина contract_churn` → агент:
1. Читает контракт
2. Запрашивает MCP структуру `ai_bi`
3. Формирует задание на витрину (шаблон 3.2)
4. Публикует в тред

### 5.2 Команда `/данные {вопрос}`

Участник пишет `/данные покажи churn за Q4` → агент:
1. Находит контракт `contract_churn`
2. Генерирует SQL по формуле из контракта
3. Выполняет через MCP `execute_sql`
4. Возвращает результат

Это уже частично реализовано в `data_query` route, но не привязано к контрактам.

### 5.3 Автоматическая сверка контракт ↔ витрина

При создании/обновлении контракта:
- Проверить, совпадает ли формула в контракте с SQL витрины
- Если расхождение — создать алерт в тред

## 6. Пример: Extra Time Dashboards → задание на витрину

На основе переписки в канале, вот как бы выглядело автоматически сгенерированное задание:

```markdown
# Задание на создание витрины: Extra Time Dashboards

## Контракт-основание
- ID: extra_time_dashboards
- Статус: Черновик (в обсуждении)
- Инициатор: @Юлия Ильчук

## Бизнес-требования

### Определение
Набор дашбордов, объединяющих P&L продукта, Unit Economics и метрику
Extra Time для продуктов Casebook, Caselook, Casebook API. Ключевая
производная метрика — стоимость генерации одного часа Extra Time.

### Формула
```
Стоимость генерации Extra Time = Total Product Cost / Total Extra Time Generated

Extra Time по продукту = Σ (Saved_Time_per_job × Jobs_done) по всем User Stories

Total Product Cost = из P&L Knowledge контракта
```

### Гранулярность
Квартальный расчёт, ежемесячное обновление.

### Включает
- P&L по продуктам (выручка, затраты, маржа)
- Unit Economics (ARPU, cost per user, MAU)
- Extra Time impact (экономия времени юристами)
- Delivery метрики (Throughput, Lead Time) — для расчёта скорости генерации

### Источники данных
| Домен | Источник | Ответственный |
|-------|----------|---------------|
| P&L | Контракт P&L Knowledge | @Юлия Ильчук |
| Extra Time атрибуты | Product инстанс (product.pravo.tech) | @Станислав Баньковский |
| Usage метрики | Продуктовая аналитика | Лидеры продуктов |
| Delivery метрики | Jira + product.pravo.tech | @Александр Первицкий |

## Технические требования

### Предлагаемые витрины (схема `ai_bi`)

**Витрина 1: dm_product_pnl** (из контракта P&L Knowledge)
```sql
-- Ежемесячный P&L по продуктам
CREATE TABLE ai_bi.dm_product_pnl (
    report_month DATE,
    product_name VARCHAR(100),  -- Casebook, Caselook, Casebook API
    revenue NUMERIC,
    cost NUMERIC,
    margin NUMERIC,
    margin_pct NUMERIC,
    loaded_at TIMESTAMP DEFAULT NOW()
);
```

**Витрина 2: dm_extra_time_by_product**
```sql
-- Extra Time по продуктам, агрегат по карточкам работ
CREATE TABLE ai_bi.dm_extra_time_by_product (
    report_month DATE,
    product_name VARCHAR(100),
    job_level VARCHAR(20),        -- jtbd / core_job / sub_job
    total_stories INT,
    stories_with_et_attr INT,     -- с атрибутом Extra Time
    total_saved_hours NUMERIC,    -- суммарная экономия
    avg_saved_per_job NUMERIC,
    loaded_at TIMESTAMP DEFAULT NOW()
);
```

**Витрина 3: dm_product_unit_economics**
```sql
-- Unit Economics по продуктам
CREATE TABLE ai_bi.dm_product_unit_economics (
    report_month DATE,
    product_name VARCHAR(100),
    mau INT,
    arpu NUMERIC,
    cost_per_user NUMERIC,
    jobs_per_user NUMERIC,
    loaded_at TIMESTAMP DEFAULT NOW()
);
```

**Витрина 4: dm_extra_time_cost** (производная)
```sql
-- Стоимость генерации Extra Time
CREATE VIEW ai_bi.dm_extra_time_cost AS
SELECT
    p.report_month,
    p.product_name,
    p.cost AS total_product_cost,
    e.total_saved_hours,
    CASE WHEN e.total_saved_hours > 0
         THEN p.cost / e.total_saved_hours
         ELSE NULL
    END AS cost_per_extra_time_hour,
    u.mau,
    u.arpu
FROM ai_bi.dm_product_pnl p
JOIN ai_bi.dm_extra_time_by_product e
  ON p.report_month = e.report_month AND p.product_name = e.product_name
JOIN ai_bi.dm_product_unit_economics u
  ON p.report_month = u.report_month AND p.product_name = u.product_name;
```

## 8. Резюме

Главные проблемы канала — **агент слишком многословен** и **не использует MCP для проактивной проработки данных**. Участники готовы работать, но тонут в статус-апдейтах. Data-команда ждёт конкретику (таблицы, поля, формулы), а агент фиксирует "нужны источники" без попытки найти их самостоятельно.

Ключевые изменения:
1. **Коммуникация**: короткие ответы, переадресация вопросов владельцу, без повторных статусов
2. **MCP-интеграция**: автоматический поиск таблиц при заполнении источника данных
3. **Автогенерация заданий**: при согласовании контракта — готовое ТЗ для Data-команды с SQL-заготовками
